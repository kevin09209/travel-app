# 30 — 派工 prompt 模板

> 用法：複製對應模板，填掉所有 `{...}` 空格後送出。**填不出「驗收條件」就代表你自己
> 還不知道要什麼，先想清楚再派工。** 模型選擇與升降級見 [10-model-dispatch.md](10-model-dispatch.md)。
> 所有模板共用的規則：subagent 是無狀態的，看不到主對話——任何它需要的背景（路徑、
> 使用者原話、先前結論）都要寫進 prompt，不要說「如前所述」。
> **model 等級一律填進 Agent 呼叫的 `model` 欄位（內聯）**：分級只寫在守則檔裡，
> 派工時會被靜默忽略。Agent 工具沒有 effort 參數，不要在呼叫裡填 effort。

## 通用尾段（每個模板結尾都附上）

```
回報格式：
- 結論：{3-5 句以內}
- 關鍵位置：檔案:行號 清單
- 驗收條件逐條核對：每條寫「過／不過＋證據」
- 超過 30 行的產物寫到 {落檔路徑}，只回傳路徑
- 若失敗：卡在哪一步、試過哪些做法、最後的錯誤訊息原文
```

> **唯讀 agent 例外**：`Explore` 與 `Plan` 不能寫檔。派它們時**刪掉落檔那一行**，改成
> 「只回 `檔案:行號` 清單＋每項一行摘要；結果超過 30 項時先回最相關的 20 項並註明總數」。
> 長產物必須落檔時，一開始就改派 `general-purpose`。

## 1. 搜尋（agent: Explore，廣度 quick / medium / very thorough）

```
在 {目錄或 repo 範圍} 找出 {目標：定義處／所有引用點／某行為的實作位置}。
動機：{主對話接下來要拿這個結果做什麼——讓它知道哪種答案有用}。
搜尋廣度：{quick=單點查找 / medium=一般 / very thorough=跨多處命名慣例}。
已知線索：{關鍵字、可能的檔名、相關模組；沒有就寫「無」}。
驗收條件：
- 每個結果附 檔案:行號 與一行說明為什麼相關
- 若找不到，列出實際搜過的 pattern 與目錄，不准只回「找不到」
```

## 2. 實作（agent: general-purpose；模式明確時 model: haiku，否則 sonnet）

```
實作：{一句話說清楚要做的功能}。
動機：{為什麼要做，這個功能給誰用}。
範圍：改 {明確的檔案清單或目錄}；**不要動** {明確排除的範圍}。
規格：{輸入、輸出、邊界情況的預期行為，越具體越好}。
遵循慣例：先看 {同類型的現有檔案} 的寫法，模仿其風格與結構。
驗收條件：
- {可機械檢查的條件 1，例：pytest tests/test_x.py 全綠}
- {條件 2，例：對空輸入回傳 []，不拋例外}
- 改動範圍不超出上述清單（git diff --stat 自查後附在回報）
```

## 3. 重構（agent: general-purpose，model: sonnet）

```
重構：{對象} —— {目標形態，例：把 X 從 Y 中抽成獨立模組}。
動機：{現在的痛點}。
不變式（重構的鐵律）：對外行為完全不變。具體指 {公開 API 簽名／輸出格式／副作用} 不得改變。
安全網：先跑 {既有測試指令} 確認基線是綠的再動手；沒有測試就先為受影響行為寫最小測試。
驗收條件:
- 動手前後測試皆綠，回報附兩次輸出
- 不新增任何功能、不順手修無關的東西；發現無關問題記下來回報，不要修
- {結構性條件，例：X.py 不再 import Y}
```

## 4. 研究（agent: general-purpose；Claude/Anthropic 相關改用 claude-code-guide）

```
研究問題：{一句話，最好是可以被證實或否證的形式}。
動機：{這個答案將決定什麼——讓它知道精度要求}。
來源要求：{官方文件優先／需要 2026 年後的資訊／允許或不允許用內建知識}。
驗收條件：
- 每個結論標註來源（URL 或「內建知識，信心高/中/低」）
- 查不到的明確寫「未能證實」，禁止用聽起來合理的話填空
- 區分「事實」與「你的推論」，分開陳列
- 完整筆記落檔 {路徑}，回報只給結論摘要
```

## 5. 審查（agent: general-purpose，fresh context；高風險用 model: opus）

```
審查對象：{檔案路徑或 diff 範圍}。
審查目的：{找正確性 bug ／ 驗收某次交付 ／ 對抗審查找矛盾}。
驗收條件清單（逐條核對）：{貼上當初派工時的驗收條件}。
額外檢查：{路徑與工具名是否真實存在／規則之間是否互相矛盾／弱模型會不會誤讀某句}。
規則：
- 你沒有參與產出，這是刻意的。不要腦補作者意圖，只看檔案實際寫了什麼。
- 每個問題附 檔案:行號、問題描述、一個具體的失敗情境（什麼輸入/情況下會出錯）。
- 找不到問題就明說「逐條核對通過」，禁止為了交差硬擠小問題。
```

## 6. 交接（HANDOFF）

觸發條件（任一成立就先落檔再繼續）：context 已被自動摘要過、即將換模型或換 session、
或使用者說「先到這裡」。落檔到工作目錄的 `HANDOFF.md`（或使用者指定路徑），六欄全填，
沒有內容的欄寫「無」，優先寫具體事實而不是感想：

```
# HANDOFF {日期}
1. 目標（使用者原話，逐字貼，不要用你摘要的版本）：
2. 當前狀態（已完成項＋各自的證據：測試輸出／read-back 結果）：
3. 已變更檔案（完整路徑清單；已備份的註明備份檔名）：
4. 決策紀錄（每條：選了什麼＋為什麼＋否決了什麼替代案）：
5. 失敗事項（試過但沒用的做法＋錯誤訊息原文——防止下一棒重踩）：
6. 精確下一步（從哪個檔案哪個動作繼續，具體到可直接執行）：
```

接手方的第一個動作：讀 HANDOFF 後逐條 read-back 核對第 2、3 欄屬實，再開工。

## 反模式（看到自己在做這些，停下來）

- 派工 prompt 只有一句「幫我看一下 X」——三件套一件都沒有。
- 把整段對話歷史剪貼給 subagent 當背景——只給它需要的，其他是雜訊。
- 同一個 agent 既做產出又做驗收——違反驗證不自驗。
- 派了五個 agent 但沒有一個有落檔路徑——結果全部湧回主對話，token 白省。
