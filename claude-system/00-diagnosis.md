# 00 — Harness 快速診斷（2026-07-03，由 Fable 5 撰寫）

> 本檔是整套制度的依據。其他檔案的規則都是針對這三個弱點的修法。
> 診斷對象：本機 Claude Code 環境（Windows 11，D:\AI 工作區），主力模型為 Sonnet 等級。

## 弱點一：主對話下場幹粗活 —— 最漏 token

**症狀**：主對話自己逐檔 Read、反覆 Grep、抓網頁全文、貼長篇 diff。每一筆原始輸出都永久佔據
context，幾輪之後觸發自動摘要（compaction），早期的關鍵約束在摘要中被沖淡，後半 session
的品質斷崖式下降。token 花在「看原料」而不是「做決策」。

**具體修法**（詳見 [10-model-dispatch.md](10-model-dispatch.md)）：
- 觸發即派工：預估要完整讀 3 個以上檔案、或同一問題 Grep 超過 2 輪、或要抓任何網頁全文
  → 一律派 subagent（搜尋用 `Explore`，其餘用 `general-purpose`），主對話只收結論與 `檔案:行號`。
- 回報合約：subagent 產出超過 30 行的內容一律落檔，只回傳路徑＋三句摘要。
- 主對話的角色是指揮官：讀結論、做決策、驗收。原料不進主對話。

## 弱點二：無驗收條件的派工與自我驗證 —— 最容易出錯

**症狀**：兩種形態。(1) 派工時只說「做什麼」，不說「做到什麼程度算完成」，subagent 回報
「已完成」但實際上檔案沒寫、測試沒跑、只改了一半。(2) 同一個 context 自己驗自己的產出——
它會用生成時的同一套假設去驗證，盲點完全重疊，等於沒驗。

**具體修法**（詳見 [10-model-dispatch.md](10-model-dispatch.md) 第 2、5 節）：
- 每次派工必含三件套：目標與動機、可機械檢查的驗收條件、回報格式。模板見
  [30-delegation-templates.md](30-delegation-templates.md)。
- 「完成」需要證據：檔案改動要 read-back（重新開 fresh agent 讀回來核對）、程式碼要測試
  輸出或實跑結果、宣稱「已修好」必須附證明。判準見 [20-judgment.md](20-judgment.md) 第 2 節。
- 高風險判斷（不可逆操作、對外發布、架構決策）加第二意見：換一個 fresh-context agent
  獨立評估，或生成多個候選答案後派評審 agent 選優。

## 弱點三：長 session 目標漂移＋經驗不落地 —— 最容易失焦

**症狀**：(1) 長 session 中 context 被壓縮後，最初的目標和使用者的口頭修正被遺忘，模型
開始優化錯誤的目標或重犯已被糾正過的錯。(2) 每個 session 踩過的坑（工具的怪癖、使用者的
偏好、環境限制）散落在對話裡，session 結束即蒸發，下一個 session 從零再踩一次。
本環境的記憶目錄在本制度建立前是空的——證明沒有制度時，經驗天然不會沉澱。

**具體修法**：
- 開工先立錨：非 trivial 任務開始時，把目標、驗收條件、不做什麼寫成三到五行清單放在回覆
  開頭；使用者每次修正方向後，立刻複述一次修正後的目標。compaction 後靠這些錨點重建方向。
- 經驗即時落地：踩坑當下（不是 session 結尾）就把教訓寫進
  [LESSONS.md](LESSONS.md)，格式見 [40-maintenance.md](40-maintenance.md) 第 3 節。
- 使用者偏好與長期事實寫入 auto-memory（`C:\Users\User\.claude\projects\D--AI\memory\`），
  注意：auto-memory 是專案綁定的，只在 D:\AI 底下的 session 生效；跨目錄通用的規則要放
  claude-system 檔案，不要只放 memory。

## 誠實條款：制度補不了的事

拆解、驗證、多樣本評審能把「執行品質」補到接近高階模型；以下兩類補不了，遇到時照辦：

1. **模糊題**（需求本身沒說清楚、多種合理解讀）：制度只能要求「停下來問」，不能替使用者
   決定。判準見 [20-judgment.md](20-judgment.md) 第 3 節。
2. **品味判斷**（架構取捨、文案語感、API 設計的優雅程度）：弱模型照 checklist 走仍可能
   做出平庸選擇。遇到時照 [20-judgment.md](20-judgment.md) 誠實條款的三選一處理；
   唯一不允許的做法是裝作有把握直接選一個。
