# 40 — 維護協議：怎麼安全地更新這套制度

> 讀者：未來任何要動 `claude-system/`、`D:\AI\CLAUDE.md` 或 `D:\AI\AGENTS.md` 的模型。
> 原則：制度可以演化，但**演化要留痕跡、可回滾、且不憑單一 session 的情緒**。

## 1. 權限分級

本節授權只適用於**本機正本**（`D:\AI\claude-system\`）。在專案副本裡（照 BOOTSTRAP.md
掛進 GitHub 專案的那份），除 LESSONS.md 外全部視為唯讀——發現錯誤記 LESSONS 並提醒
使用者回本機修正本。

**可以自行改（不必問使用者；僅限本機正本，副本一律唯讀）**：
- 在 [LESSONS.md](LESSONS.md) 追加教訓（append-only，見第 3 節格式）。
- 修正確定性錯誤：失效的路徑、改名的工具、已被官方文件推翻的參數規格。修正時在該行
  後面加註 `（2026-XX-XX 更正，原文：...）`。
- 在 auto-memory 寫入/更新使用者偏好與專案事實。

**改之前必須先問使用者（附上你的理由和建議措辭）**：
- CLAUDE.md / AGENTS.md 的鐵律七條：增、刪、放寬任何一條。
- 調度守則的閾值（幾個檔案觸發派工、重試幾輪、升降級路徑）。
- 刪除任何制度檔案或其中整個章節。
- 20-judgment.md 的任何判準——它們是刻意保守的，單一 session 覺得「太囉唆」不構成放寬理由。

**永遠不准做**：
- 無備份直接覆寫制度檔。
- 因為「這條規則在剛才那個任務裡不方便」就當場繞過。不方便 → 記進 LESSONS.md，
  由使用者決定要不要改制度。
- 手改 `claude-system/INDEX.md`——它是由 `D:\AI\CLAUDE.md` 單向生成的鏡像檔。
  要改內容就改 CLAUDE.md，再跑 `tools/sync-github.ps1` 重新生成並推送。

## 2. 修改流程（三步，不可跳）

1. **備份**：複製原檔到 `claude-system/backups/{原檔名}.{YYYY-MM-DD}.md`。根目錄入口可用
   `CLAUDE.root.{YYYY-MM-DD}.md` 或 `AGENTS.root.{YYYY-MM-DD}.md`。同一天第二次
   修改加序號 `.2`。
2. **修改**：動手改。改動幅度大時，新內容寫成新檔，原檔只留指標。
3. **read-back 驗證**：派一個 fresh agent 讀回修改後的檔案，核對：改動符合意圖、
   沒有誤刪其他段落、引用的路徑都存在。

## 3. LESSONS.md 的格式與紀律

**何時寫**：踩坑當下就寫，不等 session 結尾。三類必寫：
(a) 工具或環境的怪癖（例：某指令在 PowerShell 5.1 的行為不同）；
(b) 使用者的糾正（他說「不要這樣」的當下）；
(c) 制度規則在實戰中的摩擦（規則之間打架、閾值不合理）。

**格式**（每條四行，寫在檔案最上方，新的在前）：

```
## 2026-XX-XX {一句話標題}
- 情境：{什麼任務、什麼操作}
- 坑：{實際發生什麼}
- 之後怎麼做：{可執行的一句話}
```

**不寫**：一次性的手滑、repo 自身就記錄的事實（git history 查得到的）、與這個環境無關的通用知識。

## 4. 精簡週期

- LESSONS.md 超過 **40 條**或 **300 行**時觸發精簡：把重複主題合併、把已被吸收進制度檔的
  條目刪除（吸收＝該教訓已改寫進對應規則檔）、把過時的標記 `[已過時]` 移到檔尾。
- 精簡屬於「改制度檔」，走第 2 節三步流程。
- auto-memory 的精簡用 `/consolidate-memory` skill。

## 5. 制度健康檢查與審計重跑

**低頻健檢**：使用者說「檢查一下制度」時，派 fresh `opus` agent 做對抗審查：規則互相
矛盾、路徑失效、與當前官方文件不符的參數、LESSONS.md 裡反覆出現但沒被吸收的主題。
輸出修改建議清單，逐條附理由，由使用者裁決。
**決定性檢查**：跑 `tools/check-system.ps1` 可機械驗出行數超限與斷鏈，健檢前先跑它，
把 opus 的額度留給需要判斷的問題。
**審計重跑（觸發條件寫死）**：Claude 出了新一代最高階模型、或距上次全面審計超過六個月
——用當時最強的模型把全部制度檔重審一輪（提醒使用者：這正是回聘高階模型的正例，見第 6 節）。
模型只列問題與建議處置（檔案:行號＋問題＋提案），刪改一律由使用者決定。
上次全面審計：2026-07-05（Fable 5）。

## 6. 回聘高階模型的協議

高階模型（如 Fable/當代旗艦）額度到期後，遇到下列題型值得額外付費回聘，其餘一律用日常模型：
- **值得回聘**：對 production 服務做深度資安或架構評估、全制度審計、事關重大且
  不可逆的技術選型。回聘時它**只當診斷者與審查者**：產出可執行的改善計畫與拆單，
  執行由日常模型（sonnet/haiku）照單做。這類評估必配唯讀權限，界線靠 harness 的
  權限設定擋，優先用權限機制而不是只在 prompt 裡寫「請不要改檔案」。
- **不值得回聘**：日常 bug 卡關（先走升 opus 路徑）、機械批次工作、已有明確規格的實作。
- **切換時機**：模型切換切在計畫邊界（一份計畫做完、下一份開始前）；計畫做到一半換模型
  會丟 context 與快取，等於花錢重讀全部歷史。

## 7. 寫制度文字的口氣規則（雙讀者）

- **給弱模型的規則檔**（本目錄所有守則）：每條規則要有觸發條件、可檢查的判準、
  一正例一反例；用「優先做 X 而不是 Y」的正向句式；把該寫死的寫死。
  禁用「視情況」「用你的判斷」這類留白——弱模型接不住。
- **給強模型的任務指令**（審計、回聘時的 prompt）：只給目標與硬驗收條件，
  達成路徑留給它判斷，優先寫「要達成什麼」而不是逐步操作清單。
- 分不清讀者是誰時，按弱模型讀者寫——強模型讀寫死的規則只是囉唆，弱模型讀留白的
  規則會出錯，代價不對稱。
