# 10 — 模型調度守則

> 讀者：擔任主對話的模型（預設 Sonnet）。目的：讓主對話當指揮官，把 token 花在決策上。
> 本檔的參數規格於 2026-07-03 依官方文件查證（code.claude.com/docs）。若發現與實際 harness
> 不符，照 [40-maintenance.md](40-maintenance.md) 的流程更新本檔。

## 1. 指揮官不下場

主對話不做粗活。碰到下列任一觸發條件，立刻派 subagent，不要「先自己看一下」：

| 觸發條件 | 派誰 |
|---|---|
| 要完整讀 3 個以上檔案，或單檔超過 1000 行 | `Explore`（唯讀搜尋）或 `general-purpose` |
| 同一個問題 Grep/Glob 超過 2 輪還沒找到 | `Explore`，並指定搜尋廣度（quick / medium / very thorough） |
| 要抓任何網頁全文、查外部文件 | `general-purpose`（或 `claude-code-guide`，若問題是 Claude/Anthropic 相關） |
| 批次修改 3 個以上檔案、跑大量重複操作 | `general-purpose`，model 用 `haiku`（模式已明確時）或 `sonnet` |
| 需要規劃多步驟實作策略 | `Plan` |

例外：任務本身就是 trivial（單檔小改、一次搜尋就中），直接做，不要為派工而派工。

## 2. 派工三件套

每個派工 prompt 必含三件，缺一件就不要送出（模板見 [30-delegation-templates.md](30-delegation-templates.md)）：

1. **目標與動機**：做什麼、為什麼做（動機讓 subagent 在細節模糊時能做出正確的就地判斷）。
2. **驗收條件**：可機械檢查的完成定義。「修好 bug」不合格；「`pytest tests/test_x.py` 全綠，
   且 `foo()` 對空輸入回傳 `[]`」合格。
3. **回報格式**：明確說要回什麼、多長、什麼落檔。

## 3. 模型與 effort 的實際規格（已查證）

**Agent 工具的 `model` 參數**接受：`haiku`、`sonnet`、`opus`、完整型號 ID、或省略（省略＝
`inherit`，繼承主對話模型）。`fable` 僅在特殊授權時可用，預設不要指定。

**選型表**（平衡策略）：

| 任務性質 | model |
|---|---|
| 模式已明確的機械工作：批次改檔、格式轉換、照模板產出、簡單搜尋 | `haiku` |
| 一般實作、研究、審查、除錯（預設值） | `sonnet` 或省略 |
| 卡關升級、架構決策、對抗審查、品味判斷、高風險操作的第二意見 | `opus` |

**effort 控制的極限**（誠實條款）：
- 派工的 Agent 工具呼叫**沒有 effort 參數**，單次派工無法指定 effort。
- 可行的槓桿：(a) 在 prompt 內寫 `ultrathink` 可要求更深推理（這是唯一有效關鍵字，
  "think hard" 等已無效）；(b) 建立自訂 subagent（`.claude/agents/*.md`）時 frontmatter 可設
  `effort` 欄位，覆蓋 session 等級；(c) 使用者可在 `/model` 用左右鍵調 effort slider。
- subagent 的 extended thinking 繼承主對話設定，無法逐一指定。

## 4. 回報合約

寫進每個派工 prompt 的回報格式段落，基本款如下：

- 只回：結論（三到五句）＋關鍵位置（`檔案:行號`）＋驗收條件逐條核對結果（過／不過＋證據）。
- 超過 30 行的產物（完整報告、長 diff、抓回來的資料）一律落檔到指定路徑，回傳路徑即可。
- 失敗也要照格式回：卡在哪、試過什麼、最後的錯誤訊息原文。禁止只回「我完成了」或「有些問題」。

主對話收到回報後：不重做 subagent 的工作，只驗收。回報不合約定格式 → 退回重報一次，再犯就換人。

## 5. 升降級路徑

- **haiku 錯一次 → 直接升 sonnet**。不要對 haiku 除錯，換人比教人便宜。
- **sonnet 在同一個子任務連錯兩次 → 升 opus**，且必須附完整失敗軌跡：兩次分別怎麼做、
  錯誤訊息原文、目前的假設。不附軌跡的升級等於重新踩一次坑。
- **模式解出來之後 → 降級批次套用**：opus/sonnet 解出的做法若要重複套用多處，把做法寫成
  逐步指令，派 `haiku` 批次執行，再抽驗其中 1-2 處。
- **同一件事最多重試兩輪**。兩輪 = 同一做法的兩次嘗試；換了做法重新計數。第二輪仍失敗，
  三選一：換方法、升級模型、帶著失敗軌跡問使用者。判準見 [20-judgment.md](20-judgment.md) 第 4 節。

## 6. 驗證不自驗

產出者的 context 帶著生成時的假設，自驗會盲。驗收一律派 **fresh-context** 的 subagent（新開一個
Agent，不用 SendMessage 延續舊的）：

| 產出類型 | 驗法 |
|---|---|
| 檔案／文件 | read-back：fresh agent 讀回檔案，逐條核對驗收條件與內文一致性 |
| 程式碼 | 跑測試或實跑，貼輸出。沒有測試就先寫最小驗證腳本再跑 |
| 高風險判斷（不可逆、對外、架構） | 第二意見：fresh `opus` agent 獨立評估；或生成 2-3 個候選答案，派評審 agent 附理由選優 |

驗收 agent 的 prompt 只給驗收條件和產出位置，**不給**產出過程的推理——避免被原作者的思路帶著走。
