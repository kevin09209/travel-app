## 三件使用者沒問、但你最需要知道的事

**一、制度是槓桿，不是儀式。**
每條規則存在的理由都寫在 [00-diagnosis.md](00-diagnosis.md)：省 token、防錯、防漂移。
如果你發現自己在為一行修改派 subagent、為一句問答讀完六個制度檔，你正在把槓桿用成儀式——
這比沒有制度更糟，因為它燒掉使用者的錢還讓他以為有制度在保護他。粗略的自查線：花在
制度上的動作不該超過任務本身的一成。trivial 任務的正確做法就是直接做完。

**二、所有閾值都是推測值，不是測量值。**
「3 個檔案觸發派工」「重試兩輪」「LESSONS 滿 40 條精簡」——這些數字是我在一個還沒發生過
任何真實工作的環境裡定的，是有依據的猜測，不是實測結果。頭幾週的真實使用才是校準期：
規則造成摩擦時，把摩擦當資料記進 [LESSONS.md](LESSONS.md)，不要因為「這是 Fable 寫的」
就不敢動——但改閾值要走 [40-maintenance.md](40-maintenance.md) 的問過使用者流程。

**三、使用者的糾正是這個環境裡最貴的資料。**
他用繁體中文溝通、要求不可逆操作先確認、而且思考方式是「立制度」而非「下指令」。
這種使用者不會重複糾正你三次——第一次糾正你就要寫進 auto-memory（feedback 類型，
附 Why 和 How to apply），第二次再犯他就會開始不信任整套制度。另外記住：auto-memory
綁定 D:\AI 這個專案路徑，若他在別的目錄開工，你的記憶是空的，只有全域指標
（`~/.claude/CLAUDE.md`）和這套檔案還在。

## 這套制度最可能的四種退化，與預防法

**1. 儀式化**：規則被逐字遵守、目的被遺忘。最典型的形態是「驗證劇場」——read-back 變成
產出者自己再讀一遍、驗收 agent 收到整段產出過程的推理然後照抄結論。
*預防*：每個檔案開頭都寫了「這條規則防什麼」；驗收 agent 只給驗收條件不給推理過程
（[10-model-dispatch.md](10-model-dispatch.md) 第 6 節最後一行是特意寫的）。

**2. 規格腐化**：Claude Code 每季都在變，model 參數值、effort 機制、memory 行為都可能改。
一旦某個 session 發現制度檔寫的參數是錯的，弱模型的典型反應是**整套不信**，制度靜默失效。
*預防*：所有參數規格都標了查證日期；發現不符時的動作是「修正該行＋加註」而不是忽略整檔
（40-maintenance.md 第 1 節有授權你直接修）；使用者可定期說「檢查一下制度」觸發健檢。

**3. 膨脹**：LESSONS.md 無限追加、CLAUDE.md 被歷任 session 各塞一條「重要規則」，最後
每個 session 的開場 token 成本高到弱模型開始略讀，等於全部沒讀。
*預防*：CLAUDE.md 鐵律上限七條（要加一條就得刪一條，動鐵律要問使用者）；LESSONS.md
有精簡閾值；長內容一律放引用檔，路由表保持一頁。

**4. 靜默繞過**：某次任務裡規則「剛好不方便」，session 悄悄跳過，沒人知道。下個 session
看到前例（或什麼都沒看到），繞過變成常態。
*預防*：40-maintenance.md 明文規定「不方便 → 記 LESSONS，不准當場繞過」。你如果正在
考慮繞過某條規則，那個念頭本身就該進 LESSONS.md——寫下來的繞過是制度演化，沒寫下來的
繞過是制度死亡。
